# 项目协作规则

1. 面向用户的回复、计划、总结、验收表格、问题分析，使用中文；代码、文件名、函数名、命令行输出可以保留英文。

2. 修改前先阅读相关文件，不要凭记忆改；如果不确定当前实现，请先检查代码再判断。

3. 每次只处理用户明确要求的范围，如果注意到用户需求未明确，要主动沟通确认再做计划。不要顺手重构、顺手新增功能或改动无关文件。

4. 涉及较大改动、跨多个文件、架构调整、依赖新增、删除文件、修改核心玩法/计分/题目生成/AI 逻辑时，必须先说明方案并等待确认。

5. 修改完成后要说明：改了哪些文件、每个文件改了什么、是否影响现有功能、用户应该如何测试。

6. 如果任务涉及运行结果或界面效果，不要只根据代码判断完成；需要提醒用户实际运行/刷新/清缓存/打开控制台验证。

7. 优先保持项目简单、可读、可运行；除非用户明确要求，不要引入框架、构建工具、复杂目录结构或外部依赖。

8. 不要自动执行 git commit、git push、force push、历史重写、删除文件等操作；除非用户明确说“提交”“推送”“清理历史”或给出同等明确授权。

9. 如果需要提交代码，先给出建议的 commit message，并等待用户确认。commit message 使用简短英文 Conventional Commits 格式。

10. 不要把本地工具文件、AI 计划文件、prompt、内部规则、临时分析内容提交到仓库。这下内容应保持本地忽略

11. 对外展示内容优先保持专业、简洁、可读；README、文档、公开说明中避免出现 AI 协作痕迹、过程性记录或内部开发用语。

12. 涉及布局/几何的方案，计划中必须包含：精确的尺寸计算（px 级别的数字，不是 vague 的描述），CSS 选择器冲突检查结果，内容-容器尺寸匹配验证。
