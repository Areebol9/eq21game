## BUG #1: √(expr) / sqrt(expr) 括号开根会报"括号不匹配" ✅ 已修复 (v3.2.1)

**严重性**: P1 中度（影响 √/sqrt 的正常使用）

**用户反馈场景** (2026-04-27):
```
√(10+6)*5+5  → 括号不匹配   ← 用户实际输入报错
```

**复现矩阵** (Fuzz 测试验证, 332 个实例全部确认 ✓):
```
√(25)           → 括号不匹配
√(10+6)         → 括号不匹配
√(10+6)*5       → 括号不匹配
√(10+6)*5+5     → 括号不匹配  ← 用户场景
√(3*12)         → 括号不匹配
√(2^3)          → 括号不匹配
√(√4+7)         → 括号不匹配  嵌套 √ 一并受影响
sqrt(9)         → 括号不匹配
sqrt(16)+3      → 括号不匹配
sqrt(9+16)      → 括号不匹配
```

**根因**:
`expression.js` line 16-18,22:
- 第 16-18 行 `√` 处理：`i++` 吃掉 `√` 后，读数字→直接算 `Math.sqrt`；若遇 `(` 则生成 `TOK_SQRT` 并 `i++` 跳过 `(`，但没有额外生成 `TOK_LP`
- 第 22 行 `sqrt(` 处理：匹配 `sqrt(` 后生成 `TOK_SQRT`，`i+=5` 跳过 `sqrt(` 共 5 字符，同样没有生成 `TOK_LP`
- `toRPN()` 遇到 `)` 时在栈中找不到对应的 `(`，抛出"括号不匹配"

**影响范围**: 所有 `√(expr)` 和 `sqrt(expr)` 表达式**全部不可用**，包括嵌套 √、√+幂运算组合。
正常路径仅限 `√数字`（如 `√9`），因 tokenize 阶段直接 `Math.sqrt` 算出。

**修复方案**:
在第 18 行和第 22 行的括号消费处，各插入一个 `TOK_LP` token：
```js
// √( 分支
tokens.push({ type: TOK_SQRT, value: '√', raw: '√' });
tokens.push({ type: TOK_LP, value: '(', raw: '(' });  // ← 新增
i++;

// sqrt( 分支
tokens.push({ type: TOK_SQRT, value: 'sqrt', raw: 'sqrt' });
tokens.push({ type: TOK_LP, value: '(', raw: '(' });  // ← 新增
i += 5;
```

## BUG #2: (2+1)! 括号表达式阶乘报"非法字符" ✅ 已修复 (v3.2.1)

**严重性**: P2 低（游戏内用户很少写出这种形式）

**复现**:
```
(2+1)! → 非法字符: '!'
```

**根因**:
`expression.js` line 23-31: `!` 阶乘分支仅在数字 tokenize 后触发 (`if (i < s.length && s[i] === '!' && hasFactorial())`)。当 `!` 紧邻 `)` 时，走不到数字解析分支，落入第 44 行的默认报错。

**当前影响**: 仅 `数字!` 形式可用，`(expr)!` 不可用

**修复方案**:
在第 43 行 `)` 处理后面增加 `!` 检测：
```js
if (ch === ')') {
  tokens.push({ type: TOK_RP, value: ')', raw: ')' });
  i++;
  // 新增: 检测 ) 后面的 !
  if (i < s.length && s[i] === '!' && hasFactorial()) {
    tokens.push({ type: TOK_OP, value: '!', raw: '!' });
    i++;
  }
  continue;
}
```

或者更简单的方案：将 `!` 作为 `TOK_OP` 类型的后置运算符处理。

## BUG #3: -5! 负号阶乘报"非法字符"而非"阶乘仅支持" ✅ 已修复 (v3.2.1)

**严重性**: P3 很低（边缘场景）

**复现**:
```
-5! → 非法字符: '!'  （而非 阶乘仅支持0~20的整数）
```

**根因**:
`expression.js` line 34-38: 遇到 `-` 进入负号分支，吃掉 `5` 生成 `TOK_NUM value=-5`，但该分支不检查后续 `!`。`!` 落入第 44 行默认报错。

**当前影响**: 错误信息不精确

**修复方案**:
在负号解析后检查 `!`：
```js
if (ch === '-' && ...) {
  i++; let num = '';
  while (i < s.length && s[i] >= '0' && s[i] <= '9') { num += s[i]; i++; }
  if (!num) throw new Error('负号后需要数字');
  let n = -Number(num);
  if (i < s.length && s[i] === '!' && hasFactorial()) {
    // 阶乘在负数上无意义，直接报错
    throw new Error('阶乘仅支持0~20的整数');
  }
  tokens.push({ type: TOK_NUM, value: n, raw: '-' + num });
  ...
}
```

## BUG #4: easy/normal 难度输入 5! 报"非法字符"而非"阶乘不可用" ✅ 已修复 (v3.2.1)

**严重性**: P3 很低（体验问题）

**复现**:
```
easy:   5! → 非法字符: '!'
normal: 5! → 非法字符: '!'
```

**根因**:
`expression.js` line 23-31: 阶乘检测分支 `if (i < s.length && s[i] === '!' && hasFactorial())` 在 `hasFactorial()` 返回 false 时跳过，`!` 落入第 44 行报"非法字符"。

**当前影响**: 错误提示不够友好

**修复方案**:
在第 25 行阶乘检测处，`hasFactorial()` 为 false 时主动抛错：
```js
if (i < s.length && s[i] === '!') {
  if (!hasFactorial()) throw new Error('阶乘仅在困难模式可用');
  i++;
  ...
}
```
